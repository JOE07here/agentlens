Building AgentLens with Claude Code

A phased plan you can hand to Claude Code one step at a time — plus a ready-to-paste CLAUDE.md and the settings that keep token spend low.

For: Joe Target: Next.js + TypeScript web app Goal: thesis & portfolio build

01Before you start (one-time setup)

  1. Install a recent Node.js (LTS), then install Claude Code: npm install -g @anthropic-ai/claude-code and run claude in your project folder. (Check the official Claude Code docs for current requirements.)
  2. Make a project folder and drop your four spec docs into a /docs sub-folder (Requirements, Architecture v0.2, Data Model, Competitors). Claude Code can read them so it never re-invents the plan.
  3. Add the CLAUDE.md below to the project root.
  4. Apply the token settings in §03.

02Ready-to-use CLAUDE.md

Paste this into CLAUDE.md at your project root. It's deliberately short — a long CLAUDE.md gets re-sent on every message and quietly burns tokens.

CLAUDE.md ~40 lines · keep it lean
# AgentLens

Browser-only web app. Reads MidPoint + Keycloak exports and reports
risky non-human / AI-agent identities. Read-only. Nothing leaves the browser.

## Stack
- Next.js (App Router) + TypeScript (strict) + Tailwind
- 100% client-side in v0.1: no backend, no database, no network/API calls
- XML: fast-xml-parser. JSON: native.

## Specs — read before building a feature
- /docs has Requirements, Architecture v0.2, Data Model, Competitors
- Canonical types live in lib/model/types.ts — everything flows through them

## Hard rules
- Read-only: never write back to MidPoint or Keycloak
- No LLM, no fetch, no analytics — privacy is the product
- Keep scan data in React state (no localStorage assumptions)
- lib/correlate.ts (link MidPoint identity to its Keycloak client) is THE
  core feature — treat it with extra care

## Conventions
- TypeScript strict, no `any`
- One file per adapter and per rule; adapters output the canonical model, rules read it
- UI: high contrast only — light background, dark text. Never low-contrast.
- Small, focused components

## Build order
types → adapters → normalize → correlate → rules → score → report UI → upload wiring
Tip: if Claude can infer something from reading the code, leave it out of CLAUDE.md — and aim to keep it under 200 lines, since it loads on every message.

03Settings that keep token spend low

HabitWhat to do
Plan before codingEnter plan mode (Shift+Tab twice) for any non-trivial phase; have it write the plan, then approve. Point it at the relevant /docs file.
Right-size the modelSonnet handles most coding tasks and costs less than Opus; reserve Opus for complex work, switch with /model. Or use the opusplan alias to plan in Opus and build in Sonnet automatically.
Clear between phasesUse /clear when switching to unrelated work, since stale context wastes tokens on every later message. Start each phase fresh.
Compact mid-phaseUse /compact at a breakpoint if a phase runs long — but not after sitting idle.
Cap thinking on simple tasksFor boilerplate phases, set MAX_THINKING_TOKENS lower; keep full thinking for Correlate and the rules.
Be specificName the file and the outcome (the prompts below already do this). Vague prompts cost the most.
Watch spendRun /cost to build a feel for what each phase should cost.
The biggest saver isn't a setting — it's that you already have the specs. Front-loading the thinking (which you've done) is what stops Claude Code burning tokens rediscovering what to build. Spend your Opus budget only on Correlate and the rule logic.

04The phased build plan

Run these in order, one per session, /clear between them. Each has a ready prompt. Badges show which model to use.

0
Scaffold the app
Sonnet

Empty Next.js project with the right structure.

PromptCreate a Next.js App Router project with TypeScript (strict) and Tailwind, no backend. Read /docs/Architecture_v0.2 and create the empty folder structure from its project-structure section (lib/model, lib/adapters, lib/rules, components, config). Add empty stub files. Don't implement logic yet.
1
Canonical types
Sonnet

The data model everything flows through.

PromptIn lib/model/types.ts, implement the canonical model exactly as in /docs/Data_Model section 3: the Source, IdentityType, RiskLevel unions and the Identity, Access, Credential, Finding, and ScanResult interfaces.
2
Sample export files
Sonnet

Test data so you can build without a live system.

PromptUnder /samples, create a small MidPoint XML export and a Keycloak JSON realm export with a mix of human and non-human identities — include one service account that has an admin role and a long-lived secret, and one dormant account.
3
MidPoint + Keycloak adapters
Sonnet

Turn raw exports into the canonical model.

PromptImplement lib/adapters/midpoint.ts (parse the sample MidPoint XML with fast-xml-parser) and lib/adapters/keycloak.ts (parse the sample Keycloak JSON), each returning Identity[] with access and credentials. Add a vitest test for each against the /samples files.
4
Normalize + classify
Sonnet

Merge both sources; tag human vs non-human.

PromptImplement lib/normalize.ts to merge both adapters' output into one list and classify each identity as human or non-human using patterns in config/classification.json.
5
Correlate — the moat
Opus · plan first

Link the same agent across MidPoint and Keycloak. This is the differentiator.

Prompt (enter plan mode first)Implement lib/correlate.ts: link a MidPoint identity to its Keycloak OAuth client into one logical agent, matching by name/owner with a configurable mapping in config. Plan the matching logic and edge cases (no match, many-to-one, ambiguous names) before writing code. This is the core feature — be careful.
6
The four risk rules
Opus (logic) → Sonnet (wiring)

Over-privilege, dormancy, SoD conflict, standing credential.

PromptImplement the four rules in lib/rules/ per /docs/Requirements FR-05 to FR-08. Each takes the correlated model and returns Finding[] with a plain-language rationale and a recommendedAction. Read thresholds and the SoD conflict list from config/rules.json and config/sodConflicts.json.
7
Score & rank
Sonnet

Level each finding and sort worst-first.

PromptImplement lib/score.ts: assign each finding a risk level (high/medium/low) and a numeric score from config, then return findings ranked worst-first, plus a summary count by level.
8
Report UI
Sonnet

Show the results — clearly and in high contrast.

PromptBuild the report view: SummaryCards (counts by risk), a ranked list of FindingRow with a RiskBadge, and a per-agent cross-stack panel showing its MidPoint roles + Keycloak client + secret together. Use a high-contrast palette: light background, dark text, coloured risk badges.
9
Upload + wire it together + export
Sonnet

The end-to-end flow, all client-side.

PromptBuild the upload screen (a FileDrop for the two export files), run the full pipeline in the browser on click (ingest → normalize → correlate → rules → score), and render the report. Add lib/report/exportHtml.ts (standalone HTML) and exportData.ts (JSON + CSV).
10
Polish & real-data test
Sonnet · Opus for tricky bugs

Make it robust against messy real exports.

PromptAdd empty and error states (no file, unparseable file), then test the whole flow against a real export from demo.evolveum.com and fix any parsing edge cases.

05Definition of done (v0.1)

  • Runs entirely in the browser — no server, no upload.
  • Ingests a real MidPoint export and a real Keycloak export.
  • Correlates agents across both systems into one view.
  • Produces ranked findings, each with a plain-language reason and a recommended action.
  • Exports a standalone report (HTML/PDF) plus JSON/CSV.

Hit that, and you have your thesis artifact, your portfolio demo, and your LinkedIn proof-of-work — the three things this whole project is actually for.
